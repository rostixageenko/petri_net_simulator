# AGENTS.md — инструкции для Codex

## Роль агента

Ты работаешь над дипломным проектом «Автоматизированная подсистема моделирования и интерпретации сетей Петри». Твоя задача — помогать проектировать и реализовывать C++-ядро, Python-биндинги и JSON API для интеграции с Python/визуализационной частью.

## Важный контекст

- Это часть общего проекта. Вторая часть уже умеет визуализировать сеть и передавать описание сети Петри в JSON.
- Этот репозиторий должен принимать такой JSON, запускать моделирование и алгоритмы, возвращать JSON-результаты.
- Начальные алгоритмы: DFS, BFS, Dijkstra.
- Основной язык ядра: C++20.
- Python используется как слой интеграции и анализа.

## Приоритет реализации

1. Стабильная модель данных сети Петри.
2. Валидация входного JSON.
3. Пошаговая интерпретация сети.
4. Построение ограниченного графа достижимости.
5. DFS/BFS/Dijkstra через единый интерфейс.
6. Метрики и JSON-ответы.
7. Python-биндинги.
8. Сервисный/HTTP-адаптер, если потребуется для интеграции.

## Архитектура

Сохраняй разделение модулей:

- `core_pn` — Place, Transition, Arc, Marking, PetriNet, Interpreter.
- `graph_models` — Graph, Edge, adjacency list, FSF, BSF.
- `algorithms` — DFS, BFS, Dijkstra.
- `runtime` — реестр алгоритмов и единый протокол запуска.
- `metrics` — измерение времени, числа шагов, просмотренных вершин/рёбер, памяти при возможности.
- `bindings` — pybind11.
- `service_adapter` — JSON-входы/выходы, независимый от UI.

## Кодовые правила

- Используй C++20.
- Предпочитай простые типы и явные структуры данных.
- Не используй глобальное изменяемое состояние.
- Ошибки возвращай через `std::expected`-подобный результат, исключения или структурированный `Error`, но делай это единообразно.
- Для JSON используй `nlohmann::json`, если в проекте нет другого решения.
- Для Python-биндингов используй `pybind11`.
- Для тестов используй `Catch2` или `doctest`. Выбери один фреймворк и не смешивай.
- Добавляй тесты на каждую нетривиальную функцию.
- Не ломай публичный JSON-контракт без обновления `API_CONTRACT.md`.

## Правила для сетей Петри

- Переход разрешён, если во всех входных местах достаточно токенов с учётом веса входных дуг.
- Срабатывание перехода атомарно: сначала потребляются токены из входных мест, затем добавляются токены в выходные места.
- Дуга должна соединять место и переход. Дуги `place -> place` и `transition -> transition` недопустимы.
- Вес дуги должен быть положительным целым числом.
- В первой версии не реализовывай ингибиторные, цветные и стохастические сети, если задача явно этого не требует.

## Правила для алгоритмов

- Все алгоритмы должны иметь единый интерфейс: `run(graph, params) -> AlgorithmResult`.
- DFS/BFS должны работать на невзвешенном графе.
- Dijkstra должен работать на графе с неотрицательными весами.
- Для `reachability_graph` вес ребра по умолчанию равен `transition.fire_time`; если `fire_time` отсутствует, вес равен 1.
- Всегда возвращай метрики: время выполнения, число посещённых вершин, число просмотренных рёбер, длина/стоимость пути, статус.

## Интеграция с JSON

Поддерживай входной формат:

```json
{
  "name": "...",
  "places": [{"id": "p1", "label": "P1", "tokens": 1}],
  "transitions": [{"id": "t1", "label": "T1", "fire_time": 0.2}],
  "arcs": [{"id": "a1", "source": "p1", "target": "t1", "weight": 1}]
}
```

Выходы должны быть пригодны для визуализации: возвращай id элементов, маркировки по id мест, список сработавших переходов и ошибки в человекочитаемом виде.

## Команды проверки

После изменения кода запускай:

```bash
cmake -S . -B build -DCMAKE_BUILD_TYPE=Debug
cmake --build build
ctest --test-dir build --output-on-failure
```

Если Python-биндинги уже есть:

```bash
python -m pytest
```

## Git-правила

- Делай небольшие логические изменения.
- Не переписывай историю без явной просьбы.
- Перед коммитом проверь тесты.
- В сообщениях коммитов используй стиль: `module: short action`, например `core_pn: add transition firing`.

## Когда задавать вопрос человеку

Спроси уточнение, если:

- нужно изменить публичный JSON-контракт;
- требуется выбрать между несовместимыми архитектурными решениями;
- входные данные визуализатора отличаются от `API_CONTRACT.md`;
- тесты противоречат ожидаемому поведению сети Петри.
