# PLAN.md — план реализации

## Этап 0. Подготовка репозитория

Цель: создать каркас проекта.

Задачи:

- Настроить `CMakeLists.txt`.
- Создать структуру `include/`, `src/`, `tests/`, `examples/`, `python/`.
- Подключить `nlohmann/json`.
- Выбрать тестовый фреймворк: `doctest` или `Catch2`.
- Добавить тест загрузки `examples/mutex.json`.

Критерий готовности: проект собирается, пустой тест проходит.

## Этап 1. `core_pn` — объектная модель сети Петри

Задачи:

- Реализовать `Place`, `Transition`, `Arc`, `Marking`.
- Реализовать `PetriNet` с индексами по `id`.
- Реализовать загрузку из JSON.
- Реализовать валидацию:
  - уникальность id;
  - корректность ссылок дуг;
  - допустимое направление дуг;
  - положительный вес;
  - неотрицательное число токенов.
- Реализовать `enabled_transitions(marking)`.
- Реализовать `fire_transition(marking, transition_id)`.

Критерий готовности: mutex-сеть корректно загружается, определяются разрешённые переходы, срабатывание меняет маркировку.

## Этап 2. Интерпретатор

Задачи:

- Реализовать `Interpreter`.
- Поддержать режимы:
  - `step(transition_id optional)`;
  - `run(max_steps, strategy)`.
- Стратегии выбора перехода:
  - `first_enabled`;
  - `round_robin`;
  - `by_id`;
  - позже — пользовательская стратегия.
- Возвращать `SimulationEvent`.

Критерий готовности: можно выполнить N шагов и получить журнал событий.

## Этап 3. `graph_models`

Задачи:

- Реализовать простой `DirectedGraph`.
- Реализовать adjacency list.
- Реализовать FSF:
  - `H` — указатели начала списков смежности;
  - `B` — конечные вершины рёбер;
  - `W` — веса рёбер.
- Реализовать BSF для обратного обхода, если понадобится.
- Реализовать преобразование `PetriNet -> structural_graph`.
- Реализовать построение `reachability_graph` с ограничениями `max_states`, `max_depth`.

Критерий готовности: из mutex-сети строится граф достижимости ограниченной глубины.

## Этап 4. DFS/BFS/Dijkstra

Задачи:

- Определить `AlgorithmParams`.
- Определить `AlgorithmResult`.
- Реализовать DFS.
- Реализовать BFS.
- Реализовать Dijkstra для неотрицательных весов.
- Добавить метрики:
  - `elapsed_ms`;
  - `visited_vertices`;
  - `relaxed_edges`;
  - `path_length`;
  - `path_cost`.

Критерий готовности: алгоритмы проходят unit-тесты на малых графах и integration-тест на графе достижимости.

## Этап 5. `algorithm_runtime`

Задачи:

- Реализовать реестр алгоритмов.
- Зарегистрировать `dfs`, `bfs`, `dijkstra`.
- Сделать единый вызов: `run_algorithm(name, graph, params)`.
- Добавить ошибки для неизвестного алгоритма и некорректных параметров.

Критерий готовности: алгоритм можно выбрать строкой из JSON-запроса.

## Этап 6. JSON API / service adapter

Задачи:

- Реализовать обработчик `simulate_request`.
- Реализовать обработчик `algorithm_request`.
- Реализовать единый формат ошибок.
- Добавить примеры запросов/ответов.

Критерий готовности: клиентская часть может передать JSON и получить JSON без знания C++-структур.

## Этап 7. Python bindings

Задачи:

- Подключить `pybind11`.
- Экспортировать функции:
  - `load_net(json_str)`;
  - `simulate(json_str, params_dict)`;
  - `run_algorithm(json_str, params_dict)`;
  - `get_enabled_transitions(json_str)`.
- Добавить Python smoke tests.

Критерий готовности: Python вызывает C++-ядро и получает dict/list без ручного парсинга.

## Этап 8. Минимальный выборщик алгоритмов

Задачи:

- Реализовать простую эвристику:
  - без весов и нужна кратчайшая по числу шагов последовательность → BFS;
  - есть веса/fire_time → Dijkstra;
  - нужен полный обход/поиск любого пути → DFS.
- Сохранять историю метрик.
- Вернуть выбранный алгоритм в ответе.

Критерий готовности: `algorithm_selector` может выбрать один из трёх алгоритмов и объяснить причину.
