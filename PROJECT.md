# PROJECT.md — контекст проекта для Codex

## Тема дипломного проекта

«Автоматизированная подсистема моделирования и интерпретации сетей Петри».

Разрабатываемая подсистема является частью общего проекта. Вторая часть проекта отвечает за сбор метрик, визуализацию сети Петри и клиент-серверное взаимодействие. Эта часть отвечает за вычислительное ядро на C++ и за формальный API, через который Python/серверная часть сможет запускать моделирование и алгоритмы.

## Основная идея

Система получает описание сети Петри, параметры моделирования и выбранный алгоритм. Затем C++-ядро:

1. Создаёт внутреннее представление сети Петри.
2. Проверяет корректность структуры.
3. Выполняет интерпретацию сети: определяет разрешённые переходы, срабатывает переходы, обновляет маркировку.
4. При необходимости строит граф достижимости маркировок.
5. Запускает выбранный алгоритм: `dfs`, `bfs`, `dijkstra`.
6. Собирает метрики выполнения.
7. Возвращает результат в JSON/Python-объект для визуализации и анализа.

## Как понимать DFS/BFS/Dijkstra в контексте сети Петри

В первой версии алгоритмы можно применять в двух режимах:

### 1. Режим `reachability_graph`

Сеть Петри интерпретируется как генератор пространства состояний. Вершина графа — маркировка сети. Ребро — срабатывание перехода, которое переводит одну маркировку в другую.

- `dfs` — поиск достижимой маркировки или обход пространства состояний в глубину.
- `bfs` — поиск минимального по числу срабатываний пути от начальной маркировки к целевой.
- `dijkstra` — поиск пути минимальной стоимости, где стоимость ребра можно брать из `transition.fire_time` или из параметров запроса.

### 2. Режим `structural_graph`

JSON сети Петри преобразуется в обычный ориентированный граф элементов: места и переходы становятся вершинами, дуги сети Петри становятся рёбрами. Этот режим полезен для простого графового анализа, отладки и визуализации маршрутов.

Для диплома приоритетнее режим `reachability_graph`, потому что он показывает именно поведение сети Петри.

## Архитектурные принципы

- Ядро не должно зависеть от UI.
- Входы и выходы должны быть стабильными JSON-контрактами.
- C++-ядро должно иметь чистую библиотечную часть без HTTP, чтобы её можно было вызывать из тестов, Python и сервера.
- Слой сервиса/HTTP должен быть тонким адаптером.
- Алгоритмы должны запускаться через единый интерфейс: `run(graph, params) -> result, metrics`.
- Ошибки должны возвращаться явно: код, сообщение, контекст.
- Все изменения маркировки должны быть воспроизводимы через журнал событий.

## Основные сущности

### Place

- `id: string`
- `label: string`
- `tokens: int`

### Transition

- `id: string`
- `label: string`
- `fire_time: double`
- шаблонная C++-иерархия: `Transition<Time>`
- виртуальные/переопределяемые методы: `E`, `C`, `D`, `F`, `I`

### Arc

- `id: string`
- `source: string`
- `target: string`
- `weight: int`
- допустимые направления: `place -> transition`, `transition -> place`

### Marking

Упорядоченный вектор токенов по внутреннему индексу мест. Для JSON возвращать также map по `place.id`.

### SimulationEvent

- `step`
- `time`
- `fired_transition`
- `marking_before`
- `marking_after`
- `enabled_before`
- `enabled_after`

### AlgorithmResult

- `algorithm`
- `status`
- `path`
- `visited_count`
- `distance` или `cost`
- `metrics`

## Не делать в первой версии

- Не реализовывать полноценную распределённую систему.
- Не усложнять плагинами пользовательских алгоритмов до появления стабильного интерфейса встроенных алгоритмов.
- Не добавлять тяжёлую СУБД в первый вертикальный срез, если JSON/CSV достаточно.
- Не привязывать ядро к конкретному визуальному интерфейсу.

## Definition of Done для первого среза

Код считается готовым для первого этапа, если:

- пример `examples/mutex.json` загружается без ошибок;
- тесты проверяют разрешённость переходов для mutex-сети;
- пошаговая симуляция корректно меняет токены;
- `bfs` на графе достижимости находит путь к заданной маркировке;
- `dijkstra` использует `fire_time` как стоимость срабатывания;
- результат возвращается в согласованном JSON;
- сборка и тесты запускаются одной командой через CMake/CTest.
